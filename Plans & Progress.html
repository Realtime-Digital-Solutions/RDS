Plans & Progress

Introduction

RDS’s goal is to devise and develop a wearable medical device that can transmit vital signs of a patient, via WIFI, to be remotely monitored by medical staff. This enables medical staff to act upon a medical emergency if the patient is in danger, even if the patient is unaware.

To develop this device will require a multitude of steps and goals to be achieved on a regular basis. These steps will be broken down into smaller, more manageable steps and goals. This will help our team have clear, achievable goals each week and stop us from feeling overwhelmed.

Step 3

Development of the firmware/hardware was performed by the Lead Software Developer. Research of the hardware/software tools was performed by our Lead Research team member. We chose to use the [1] ESP8266 and the Maxim heartrate sensor for its ease of use, cost, documentation, and availability. The Arduino community have already designed this type of sensor/WiFi project before. This was another advantage for us to use this hardware as we could borrow some of the ideas the community used and implement it on our project. This saves time and effort to produce a working model.

Hardware development was simple and required a connection from the oximeter to the ESP8266. This was in the form of a I2C serial interface and power connections. The power to the ESP8266 is from the USB programming cable from the PC however, this device is intended to be powered from a battery.

Firmware development was primarily done in the Arduino IDE using the C programming language. There are many examples and pre-written libraries for us to use to keep the development time to a minimum. The libraries used are in the code snippet below from Arduino IDE. 

The Arduino libraries:

The ESP8266WiFi.h library is for handling the WiFi connection and client control.

The Wire.h library handles the I2C connection and address/command control.

The MAX30105.h library is for the Maxim sensor and is specific for that sensor series to control the data flow and commands.

The heartrate.h library is the algorithm that converts the sensor output form an arbitrary value to beats per minute.

To initialise the WiFi, serial communications and oximeter, the following code snippet was used. We used the examples in the Arduino IDE for this section of code to save time as writing the code from scratch would have been a difficult and time-consuming task.

The main (while 1 loop) section of the code (below diagram) polls the sensor, converts the value to beats per minute and connects to the simple Python server to transfer sensor data. The code works reasonably well however, when the client side is connecting to the server, there can be a delay of about 3 seconds. In that connection time, there is no sensor conversion or data collection as the connection to the server blocks its task. A timer interrupt and interrupt service routine to collect data from the sensor needs to be implemented as the sensor data is the most important data to be collected and converted.

Most of the C code was from the examples that were provided in Arduino IDE and the sensor C code example was provided by Maxim. This code is only for a basic prototype and needs to be completely re-written if this was to be implemented commercially.

Issues with the IP address of the access point need to be developed further. If the Access point/s change their IP address, this can break the connection from the ESP8266 sensor and the WiFi to the Python socket server. A more generic/common IP address needs to be implemented.

The Maxim oximeter sensor is very fussy with the placement of your finger. The values can be somewhat unreliable and constantly changing. The sensor is cheap and at $8.00, would not be suitable for a commercial application. Searching the Arduino forums, others are experiencing similar problems with device.  

Step 4

The below diagram is PyCharm IDE running a simple socket server bound to the PC WiFi interface. The second window that is displaying GET DATA(x,x,x,x) is the data from the ESP8266 which includes the heartrate, MAC address, RSSI and Boolean for finger detection. The data is in the form of a CSV, can be split up and sent to each section of the Python QT GUI. We had issues, initially with the data = socket.recv(1024) data buffer. The data buffer needs to be completely emptied before other tasks would continue, it blocked the flow of data from the ESP8266. This was undesirable. Our intentions where for the data flow from the ESP8266 to be a constant stream that is feed to the GUI for continual updates. After some research we discovered a Python module called [2] Select. The Select allowed efficient I/O from the WiFi interface to constantly stream the data from the ESP8266. The code for Select was derived from the Select module online documentation example.

The below information, in blue, is the streamed data collected from the ESP8266 client. The data in the parenthesis is the data we will use to update our GUI application. The data is separated by a comma (CSV). The first piece of data is the heartrate in BPM, the second is the MAC address, third is the RSSI level and lastly is the presence of a finger on the sensor, 0 for no finger and 1 for finger placed on sensor. The below data is detecting no finger and no heartrate.

GET DATA(0.75,48:3F:DA:76:21:85,-37,0) HTTP/1.1

Host: 192.168.43.234

Connection: close

The data below (in blue) now contains the heartrate, and a finger is detected. It can take up to 1 minute for the heartrate to settle and provide a stable value. This delay will need to be considerably shortened if used in a commercial application.

GET DATA(73.26,48:3F:DA:76:21:85,-40,1) HTTP/1.1

Host: 192.168.43.234

Connection: close

Step 5

The UI/GUI development started with [3] QT Designer in a drag and drop development environment for quick and easy GUI layout. This is an open-source software package and is free to download. Below is a snapshot of the development in progress. It was extremely easy to use and somewhat intuitive. To design our simple GUI took less than 1 hour. The conversion to Python code was not so simple.

There were some issues creating the .py Python files from QT Designer XML files. We had to find the pyuic5.exe, create a folder, save all the QT Designer files to it and then run pyuic5.exe in the folder to create the Python code. This had to be done on the command line as there is no click and convert feature in this IDE. It is quite clunky for a supported and popular GUI IDE. We may automate this process with Python in future iterations.

The GUI Python file is now a .py file that can be edited in PyCharm IDE. Our assumption was that the GUI update will be in the main thread and to run other tasks will require some sort of worker thread to take care of the socket server, data manipulation and data collection from ESP8266 for display on the GUI. Using simple threading is not possible as the main thread is the GUI, and other more taxing worker threads will block the update of the GUI and make the user experience unpleasant.

With a little research we decided to use the [4] QThread module which is suited to this application and is a recommendation for PyQt5 GUI threading. This module allows send and emit signals between worker threads and the main thread. This helps with control of the threading time slots and avoids the GUI hanging or responding slowly. Below is the application running and a worker thread with a while 1 loop that constantly prints to the console/command line. 

As you can see the GUI is visible, populated and the worker thread is printing to the console/command line. This worker thread is where the server and data collection will be executed, and variables will be written to the GUI.

The GUI will have a maximum and minimum heartrate limit that can be set by the user. When these limits are reached, ‘Emergency’ will be displayed on the GUI. If the software detects that the patient’s finger is not on the sensor, the GUI will then display “No Finger” in red text, however, ‘Emergency’ will not be displayed.  

Step 6

Integration of the GUI and the socket server will need to be implemented. The socket server will reside in the worker thread as it constantly needs to be receiving data from the ESP8266 client and updating the GUI. The program will be one Python file only and not split up into other files such as GUI, server, worker thread. The reason for this is to produce a quick and dirty prototype without dependencies. If this was developed as a commercial product, we would spend the time and effort to plan the GUI and other aspects of the PC application with careful consideration. A professional installation package would also be designed for ease of installation and to ensure proper setup of Windows environment for supporting our application. For this application, an executable file will be developed. The integration of the various code fragments wasn’t too difficult. Careful planning where the code fragments go and how to implement was a simple logic process.

The RSSI only serves the purpose of an indication for the operator to identify if the device is out of range for quality heartrate data. This RSSI indication would change colour to red if the patient was out of range.

There are some non-functioning aspects of the GUI that we didn’t have time to develop such as the time of arrival text and information. Only one device can be displayed on the GUI when the server can take up to 10 connections from individual devices. The RSSI level would turn red if the patient was out of range. The MAC address of the device is meaningless to the operator, and we would have liked to associate the MAC address with a patient’s name instead. Easier for the operator to understand. The emergency alarm and flashing screen is one function that needs to be implemented however, we have unfortunately, run out of time.
